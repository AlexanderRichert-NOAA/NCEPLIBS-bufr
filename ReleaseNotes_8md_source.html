129 * A bug was fixed in the bitmap processing, so that each Table A mnemonic in

378 ### Version 10.2.0 - October 19, 2012
379 
380 * The makebufrlib.sh script was modified to streamline the endianness check
381 and make it more portable.
382 
383 * Subroutine wrtree() was modified to ensure that "missing" character strings
384 are properly encoded with all bits set to 1.
385 
386 * A new function icbfms() was added which tests whether decoded character
387 strings are "missing" by checking if all of the equivalent bits are set to 1.
388 This was done because, on certain platforms, the BUFRLIB REAL*8 "missing" value
389 BMISS is not always equivalent to all bits set to 1 when viewed as a character
390 string, and thus the existing BUFRLIB function ibfms() did not always work
391 properly in such cases. However, users can continue to use the existing IBFMS
392 function in application programs, because the new icbfms() function has now been
393 incorporated internally within the logic of many BUFRLIB subroutines, in
394 addition to also being available for direct calling by application programs.
395 
396 * Subroutines readmg() and readerme() were modified to prevent the BUFRLIB from
397 internally adjusting to DX (dictionary) table messages when Section 3 decoding
398 is being used. Otherwise, contention can occur between the table information
399 in the DX messages and the table information specified within the Section 3
400 descriptors. From now on, whenever Section 3 decoding is used (as specified
401 by setting IO="SEC3" when opening a file via openbf()), the BUFRLIB will now
402 treat any DX (dictionary) table message the same as any other message and
403 decode the actual data (i.e. table) values according to Section 3.
404 
405 * Subroutine openbf() was modified to allow a new option for input call
406 argument IO. If this argument is set to 'INUL', then the BUFRLIB will behave
407 the same as when IO='IN', except that it will never try to actually read
408 anything from the file attached to input call argument LUNIT. This can be
409 useful for some special cases, such as when the user plans to pass input
410 messages to the BUFRLIB using subsequent calls to subroutine readerme().
411 
412 * A new subroutine gettagpr() was added which returns the mnemonic corresponding
413 to a parent sequence in a subset definition, given the mnemonic corresponding
414 to a child descriptor within that sequence. This can be useful in certain
415 application codes, especially when Section 3 decoding is being used.
416 
417 * A new function getvalnb() was added which searches for a specified mnemonic
418 in a subset definition, then searches forward or backward from that point for
419 a different mnemonic and returns the associated value. This can be useful in
420 certain application codes, especially when Section 3 decoding is being used.
421 
422 * Functionality was added to improve the portability of reading and writing
423 BUFR messages across different platforms. All calls to existing FORTRAN
424 subroutines which read or write BUFR messages from disk (e.g. readmg(), ufbmem(),
425 ufbtab(), writsb(), wrcmps(), copymg(), etc.) now use embedded C-language I/O to
426 perform these tasks. Among other things, this means that any BUFR file can
427 now be read regardless of whether it has been pre-blocked with FORTRAN
428 control words using the cwordsh utility. For writing BUFR files, a new
429 subroutine setblock() was added which allows users to specify whether output
430 BUFR messages are to be unblocked (which is the new default), big-endian
431 blocked, or little-endian blocked.
432 
433 * A new subroutine setbmiss() was added which allows users to specify a custom
434 "missing" value for writing to and reading from BUFR files, rather than using
435 the BUFRLIB default "missing" value of 10E10. A corresponding function
436 getbmiss() was also added which returns the current "missing" value in use.

1626 
1627 <u>Y2K Compliance</u>
1628 
1629 Y2K compliance in the BUFR Archive Library is downwardly compatible. That is,
1630 the new library will read non-Y2K BUFR files as the old one does. However, all
1631 two digit years read are represented internally as four digit years, and any
1632 files written with the new library will be in Y2K format. A functional
1633 conversion of two digit year inputs assumes the years 21-99 are in the
1634 twentieth century, while years numbered 00-20 are in the twenty-first. A Y2K
1635 BUFR file is identified by a non-zero value in the 18th byte of the message
1636 section one, the century byte. At this point users of the library have access
1637 to the full four digit year values read by including a signal in their programs
1638 via a new entry point called DATELEN. The plan is to have the default set to
1639 return two digits of the year during a transition period. This allows
1640 implementation of the new BUFR Archive Library into a non-Y2K compliant
1641 environment. The susbsequent list of subroutines have been changed for Y2K
1642 compliance: BFRINI, DATEBF, DUMPBF, MSGINI, OPENMB, OPENMG, RDMEMM, READERME,
1643 READFT, READMG, READTJ.
