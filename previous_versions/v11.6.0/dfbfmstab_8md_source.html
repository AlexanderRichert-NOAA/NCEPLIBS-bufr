2 ## Master BUFR Tables
3 @brief Description and format of master BUFR tables files for use with the library
4 @anchor dfbfmstab
5 
6 This document describes the format and concept of master BUFR tables. These are required by the BUFRLIB software
7 whenever the IO='SEC3' option is specified during a call to subroutine openbf() for the reading/decoding of a
8 given file of BUFR messages. Otherwise, if a different value of IO is specified, then only a
9 [DX BUFR tables](@ref dfbftab) file is normally required, and master BUFR tables are not needed.
10 
11 <br>
12 Whenever master BUFR tables are used, they are read in by the BUFRLIB software as a corresponding set of four
13 system files, from a directory on the local filesystem as specified within a separate preceding call to
14 subroutine mtinfo() :
15 
16 - <i>Standard Table B</i> - an ordered listing of all descriptors corresponding to an official release
17 (i.e. version number) of the standard, international BUFR Table B
18 - <i>Local Table B</i> - an ordered listing of all local Table B descriptors supplementing the standard
19 BUFR Table B, as defined for use within a particular local originating center
20 - <i>Standard Table D</i> - an ordered listing of all descriptors corresponding to an official release
21 (i.e. version number) of the standard, international BUFR Table D
22 - <i>Local Table D</i> - an ordered listing of all local Table D descriptors supplementing the standard
23 BUFR Table D, as defined for use within a particular local originating center
24 
25 Actual filenames use the following convention:
26 
27 <pre>
28 Standard tables: bufrtab.Table.<i>X</i>_STD_<i>M</i>_<i>V</i>
29 Local tables: bufrtab.Table.<i>X</i>_LOC_<i>M</i>_<i>C</i>_<i>L</i>
30 
31  where: <i>X</i> = Table type ('B' or 'D')
32  <i>M</i> = Master Table number ('0' for WMO, '10' for IOC, etc.)
33  <i>V</i> = Version number of Master Table used
34  <i>C</i> = Originating Center number ('7' for NCEP, etc.)
35  <i>L</i> = Version number of local tables used
36 </pre>
37 
38 Whenever any of the [message-reading subroutines](@ref hierarchy) are used to read a new BUFR message
39 from a Fortran logical unit that was previously opened using subroutine
40 openbf() with IO='SEC3', the identification section (Section 1) of the message is automatically scanned to
41 determine the above values for that message. The BUFRLIB software then automatically generates the four necessary
42 filenames using those values and attempts to open and read each of those files from within the directory that was
43 specified as CMTDIR during the previous call to subroutine mtinfo(). This table information is then retained
44 and re-used in memory until a subsequent message is read which has a different value for any one of the
45 above values, and at which point a new set of tables is then read in from the same directory in order to be
46 applied to the new message. With this approach, the user can have multiple master table files stored within
47 the same directory, and the BUFRLIB software will always locate and read the appropriate files depending on
48 the corresponding values stored within Section 1 of each new message to be decoded. For more details about
49 the above values, see the discussion on Section 1 within the official
50 [WMO Manual 306, Volume I.2](@ref manual).
51 
52 <br>
53 Now that we've discussed the content and naming conventions for master BUFR table files, let's turn our
54 attention to the actual format of these files:
55 
56 <div id="TableB"/>
57 ### Table B
58 
59 As described above, two master Table B files (one standard and one local) are required for each BUFR message
60 that is to be read and decoded. The BUFRLIB software will scan the identification section (Section 1) of
61 each new message and then open and read the appropriate Table B files from within the directory specified by
62 CMTDIR during the previous call to subroutine mtinfo(). In particular, a standard Table B file must be
63 available corresponding to the version number of each BUFR message to be decoded, and the filename must
64 follow the naming convention described above. In addition, if there are any local descriptors contained
65 within the messages, then a local Table B file must also be available corresponding to the originating center
66 and local version numbers encoded within Section 1 of those messages. Otherwise, if the messages to be
67 decoded contain only standard descriptors (which is normally the case for data exchanged between operational
68 centers), then the
69 [default local Table B file from NCEP (originating center 7)](https://github.com/NOAA-EMC/NCEPLIBS-bufr/tree/develop/tables/bufrtab.TableB_LOC_0_7_1)
70 can be used as a placeholder
71 since the software will not need to actually read any information from this table. In this way, users are
72 relieved from having to provide a local Table B file for every originating center whose messages they wish
73 to decode, and such a table only becomes necessary when the messages themselves actually contain one or more
74 local descriptors defined by that particular originating center.
75 
76 <br>
77 
78 Here now is the format for each master Table B:
79 
80 <br>
81 
82 The first line of the file is as follows, where the symbols correspond to those used in the file naming
83 convention described above. In this way, the software can perform internal consistency checks where needed.
84 Note that the separator for each field is a "|" character, but otherwise the free use of white space is
85 allowed within each field:
86 
87 <pre>
88 Standard table: Table B STD | <i>M</i> | <i>V</i>
89 Local table: Table B LOC | <i>M</i> | <i>C</i> | <i>L</i>
90 
91  where: <i>M</i> = Master Table number ('0' for WMO, '10' for IOC, etc.)
92  <i>V</i> = Version number of Master Table used
93  <i>C</i> = Originating Center number ('7' for NCEP, etc.)
94  <i>L</i> = Version number of local tables used
95 </pre>
96 
97 Following this first header line, the remainder of the file contains a listing of all possible Table B
98 descriptors corresponding to the specified version number of the table, ordered in ascending order with
99 respect to the FXY number as shown below. Blank lines as well as comments (i.e. any line beginning with
100 the character "#") are permitted throughout the file and will be ignored by the software, up until a line
101 is reached which contains the string "END" in the first three characters, and which is a signal to the
102 software to stop looking for any new Table B descriptor definitions in the table. Each definition line
103 has the following format, where the "|" and ";" characters are required separators as shown, but
104 otherwise the additional use of white space is allowed within each field.
105 
106 <pre>
107  F-XX-YYY | SCALE | REFERENCE | BITS | UNITS | MNEMONIC ; CODES ; ELEMENT NAME
108 </pre>
109 
110 The CODES field can be used for any annotations the user may wish to add for a particular definition line,
111 or it can be left blank. All other fields should be self-explanatory based on earlier discussions and
112 should contain a value. Here are some examples:
113 
114 <pre>
115  0-01-018 | 0 | 0 | 40 | CCITT IA5 | SSTN ; ; Short station or site name
116  0-01-041 | 5 | -1073741824 | 31 | m/s | PS00 ; ; Absolute platform velocity - first component
117  0-05-002 | 2 | -9000 | 15 | Degree(N+,S-) | CLAT ; ; Latitude (coarse accuracy)
118  0-07-002 | -1 | -40 | 16 | m | HMSL ; ; Height or altitude
119 </pre>
120

122 to individual preferences, but of course for overall readability it's usually best to pick one format

153 ### Table D
154 
155 As was the case for [Table B](#TableB), two master Table D files (one standard and one local) are also
156 required for each BUFR message that is to be read and decoded, and these files must also exist within
157 the same directory specified by CMTDIR during the previous call to subroutine mtinfo(). The BUFRLIB
158 software will then scan the identification section (Section 1) of each BUFR message as it is read, in
159 order to determine the exact master table files to open and read for that message, and a local Table D
160 file is necessary whenever local descriptors from the originating center in question are included
161 within a message; otherwise, the
162 [default local Table D file from NCEP (originating center 7)](https://github.com/NOAA-EMC/NCEPLIBS-bufr/tree/develop/tables/bufrtab.TableD_LOC_0_7_1)
163 can be used as a placeholder.
164 
165 <br>
166 
167 The format of the first line of each master Table D file is the same as for Table B:
168 
169 <pre>
170 Standard table: Table D STD | <i>M</i> | <i>V</i>
171 Local table: Table D LOC | <i>M</i> | <i>C</i> | <i>L</i>
172 
173  where: <i>M</i> = Master Table number ('0' for WMO, '10' for IOC, etc.)
174  <i>V</i> = Version number of Master Table used
175  <i>C</i> = Originating Center number ('7' for NCEP, etc.)
176  <i>L</i> = Version number of local tables used
177 </pre>
178 
179 and blank lines and comment lines (beginning with a "#" in column 1) are likewise allowed, but
180 otherwise the format for a master Table D file differs from that of a master Table B file since
181 entries now span multiple file lines as follows:
182 
183 <pre>
184  F-XX-YYY | MNEMONIC ; CODES ; SEQUENCE_NAME
185  | F-XX-YYY > | ELEMENT_NAME
186  | F-XX-YYY | ELEMENT_NAME
187 </pre>
188 
189 As shown, the first line of each entry contains the FXY number of the Table D sequence, along with the
190 corresponding sequence name, mnemonic and annotation codes (if any), and then each successive line
191 contains a single element of that sequence. Each successive element is listed one per line, and note
192 the ">" character after the FXY number within each element line up to, but not including, the last
193 element of the sequence. The BUFRLIB software uses the lack of a ">" character to know when it has
194 reached the last element of the sequence. Here are some examples:
195 
196 <pre>
197  3-01-004 | SFCSTNID ; ; Surface station identification
198  | 0-01-001 > | WMO block number
199  | 0-01-002 > | WMO station number
200  | 0-01-015 > | Station or site name
201  | 0-02-001 | Type of station
202 
203  3-01-012 | HHMM ; ;
204  | 0-04-004 > | Hour
205  | 0-04-005 | Minute
206 
207  3-01-023 | LTLONC ; ;
208  | 0-05-002 > | Latitude (coarse accuracy)
209  | 0-06-002 | Longitude (coarse accuracy)
210 
211  3-01-025 | LTLONCDT ; ;
212  | 3-01-023 > | Latitude and longitude (coarse accuracy)
213  | 0-04-003 > | Day
214  | 3-01-012 | Time
215 
216  3-01-045 | SATLOVEL ; ; Satellite location and velocity
217  | 3-01-011 > | Year, month, day
218  | 3-01-012 > | Time (hour, minute)
219  | 2-01-138 > | Change width to 16 bits
220  | 2-02-131 > | Change scale to 3
221  | 0-04-006 > | Second
222  | 2-01-000 > | Change width back to Table B
223  | 2-02-000 > | Change scale back to Table B
224  | 3-04-030 > | Location relative to the Earth's centre
225  | 3-04-031 | Velocity relative to the Earth's centre
226 
227  3-03-050 | WDPLRAOB ; ; Wind data at a pressure level with radiosonde position
228  | 0-04-086 > | Long time period or displacement (since launch time)
229  | 0-08-042 > | Extended vertical sounding significance
230  | 0-07-004 > | Pressure
231  | 0-05-015 > | Latitude displacement since launch site (high accuracy)
232  | 0-06-015 > | Longitude displacement since launch site (high accuracy)
233  | 0-11-001 > | Wind direction
234  | 0-11-002 | Wind speed
235 </pre>
236

270 ### Code/Flag Tables
271 
272 Unlike for [Table B](#TableB) and [Table D](#TableD), master Code/Flag tables are optional when
273 the IO='SEC3' option is specified during a call to subroutine openbf() for the reading/decoding
274 of BUFR messages. Instead, they are only required if the user intends to make one or more
275 calls to subroutine getcfmng(), and in which case a prior call to subroutine codflg() is also
276 required with the value of CF set to 'Y'.
277 
278 <br>
279 
280 Whenever master Code/Flag tables are used, they must exist within the same local filesystem
281 directory specified by CMTDIR during the most recent call to subroutine mtinfo(). And just like
282 for Table B and Table D, they must exist as a set of two separate tables files, one containing
283 all of the standard entries and one containing all of the local entries, and where the
284 [default local Code/Flag tables file from NCEP (originating center 7)](https://github.com/NOAA-EMC/NCEPLIBS-bufr/tree/develop/tables/bufrtab.TableF_LOC_0_7_1)
285 can be used as a placeholder for the local file whenever the BUFR messages to be decoded contain
286 only standard descriptors.
287 
288 <br>
289 
290 The format of the first line of each master Code/Flag table file is the same as for Tables B and D:
291 
292 <pre>
293 Standard table: Table F STD | <i>M</i> | <i>V</i>
294 Local table: Table F LOC | <i>M</i> | <i>C</i> | <i>L</i>
295 
296  where: <i>M</i> = Master Table number ('0' for WMO, '10' for IOC, etc.)
297  <i>V</i> = Version number of Master Table used
298  <i>C</i> = Originating Center number ('7' for NCEP, etc.)
299  <i>L</i> = Version number of local tables used
300 </pre>
301 
302 and blank lines and comment lines (beginning with a "#" in column 1) are likewise allowed
303 throughout each table for readability. Entries span multiple lines and follow the format:
304 
305 <pre>
306  F-XX-YYY | MNEMONIC ; CODE
307  | F-XX-YYY=VAL
308  | VAL > | MEANING
309  | VAL | MEANING
310 </pre>
311 
312 for code tables, and:
313 
314 <pre>
315  F-XX-YYY | MNEMONIC ; FLAG
316  | F-XX-YYY=BIT
317  | BIT > | MEANING
318  | BIT | MEANING
319 </pre>
320 
321 for flag tables. As shown, the first line of each entry contains the FXY number and mnemonic
322 of the associated Table B descriptor, along with the identifier "CODE" or "FLAG" depending on
323 the table type. Successive lines contain individual entries within the associated table,
324 grouped by dependency where applicable. For example, for mnemonic GSES (originating sub-center),
325 whose meanings depend on that of the associated originating center as noted in the discussion
326 for subroutine getcfmng(), the entry would look something like the following, and where 0-01-031,
327 0-01-033 and 0-01-035 are the respective FXY numbers for mnemonics GCLONG, OGCE and ORIGC:
328 
329 <pre>
330  0-01-034 | GSES ; CODE
331  | 0-01-031,0-01-033,0-01-035=39
332  | 0 > | No sub-centre
333  | 225 > | Beijing
334  | 226 > | Guangzhou
335  | 228 > | Urumuqi
336  | 0-01-031,0-01-033,0-01-035=80
337  | 0 > | No sub-centre
338  | 101 > | Albania (NMC)

340  | 0-01-031,0-01-033,0-01-035=7
341  | 0 > | No sub-centre
342  | 1 > | NCEP Reanalysis Project
343  | 2 > | NCEP Ensemble Products
344  | 3 > | NCEP Central Operations
345  | 4 > | Environmental Modeling Center
346  | 5 > | Weather Prediction Center
347  | 6 > | Ocean Prediction Center
348  | 7 > | Climate Prediction Center
349  | 8 > | Aviation Weather Center
350  | 9 > | Storm Prediction Center
351  | 10 > | National Hurricane Center
352  | 11 > | NWS Techniques Development Laboratory
353  | 12 > | NESDIS Office of Research and Applications
354  | 13 > | Federal Aviation Administration
355  | 14 > | NWS Meteorological Development Laboratory
356  | 15 > | North American Regional Reanalysis Project
357  | 16 > | Space Weather Prediction Center

443 the ">" character after the value within each line up to, but not including, the last defined

445 know when it has reached the last value/meaning pair for the associated entry. And as was also
446 the case for master Table B and Table D files, all entries in a master Code/Flag table file
