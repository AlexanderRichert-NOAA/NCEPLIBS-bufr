483 As noted during the discussion of subroutine openbf(), every BUFR file that is presented to the
484 BUFRLIB software, either for input (reading/decoding) or output (writing/encoding) purposes, must
485 have DX BUFR tables associated with it, unless the 'SEC3' decoding option is specified during the
486 call to openbf(). For all other cases, DX table information must be pre-defined and made available
487 to the software via call argument LUNDX during the call to openbf(). In the case of an existing
488 BUFR file, the DX tables information may be embedded within the first few BUFR messages of the
489 file itself. Otherwise, a separate ASCII text file containing the necessary DX tables information
490 must be supplied, such as the example shown above. It is extremely important that any such file
491 not only be syntactically correct but also complete, in the sense that all necessary mnemonics
492 must exist and be fully-defined.
493 
494 <br>
495 
496 First, let's define what we mean by a <i>mnemonic</i>. In short, a mnemonic is simply a
497 descriptive, alphanumeric name for a data value. In the context of the BUFRLIB software, there
498 are "Table A mnemonics", which refer to particular data subset (i.e. report ) types, "Table B
499 mnemonics", which refer directly to basic data values, and "Table D mnemonics", which are
500 sequences composed of one or more Table B (or other Table D) mnemonics and which are themselves
501 normally direct constituents of a particular Table A mnemonic. In other words, at the highest level,
502 we have a Table A mnemonic which completely describes a type of data subset (e.g. rawinsonde,
503 wind profiler, etc.), and this Table A mnemonic is defined as a sequence of one or more Table B or
504 Table D mnemonics, where each Table D mnemonic is likewise itself defined as a sequence of one or
505 more Table B or Table D mnemonics, and so on until the entire data subset can be equivalently
506 described as a sequence of one or more Table B mnemonics which, again, themselves correspond to
507 basic data values (e.g. pressure, temperature, humidity, etc.). In this way, the entire sequence
508 of data values that constitute a particular type of data subset is fully and unambiguously defined,
509 both for purposes of input (reading/decoding) or output (writing/encoding) of reports corresponding
510 to that particular type of data subset.
511 
512 <br>
513 
514 However, it's also important to understand what mnemonics are not. Specifically, mnemonics never
515 themselves appear within actual BUFR messages that are read or written by the BUFRLIB software;
516 rather, their only purpose in life is to make it easier for users to interact with the software by
517 providing descriptive names to represent individual data values, as opposed to having to keep track
518 of the corresponding FXY numbers (described below), which are much less intuitive but which
519 nevertheless are the prescribed method within the BUFR code form for referencing of individual data
520 values, and which therefore are what are actually read and written by the software.
521 
522 <br>
523 
524 Before continuing on let's recall
525 that a DX BUFR tables file consists of three distinct sections. Each section contains one or more
526 lines of 80 characters in length, and where a "*" as the first character of a line indicates that
527 that entire line is a comment. In the [first section](#section1), all Table A, B and D mnemonics that are to be
528 used within the file are initially declared, assigned a unique FXY number, and given a short,
529 free-form text description. Then, in the [second section](#section2), all previously-declared Table A and
530 Table D mnemonics are actually defined as a sequence of one or more Table B (or other Table D)
531 mnemonics. Finally, in the [third section](#section3), all previously-declared Table B mnemonics are defined
532 in terms of their scale factor, reference value, bit width, and units.
533 
534 <br>
535 
536 Now, as we delve into the details of each of the three sections, we'll constantly refer back to
537 our [sample DX BUFR tables file](#bftab) in order to better illustrate the concepts that are discussed.

539 <div id="section1"/>
540 ### Section 1
541 
542 As previously mentioned, the first section of a BUFR tables file is where all Table A, B and D
543 mnemonics are initially declared, assigned a unique FXY number, and given a short free-form text
544 description. Mnemonics may contain any combination of uppercase letters and numbers (or, in certain
545 special cases, a "." character!), up to a maximum total of 8 characters in length. A mnemonic may be
546 declared only once, and each one must correspond to a unique FXY number, which itself consists of 6
547 characters, and where the first character (i.e. the "F" component) is an "A" if the mnemonic is
548 being declared as a Table A mnemonic, "3" if the mnemonic is being declared as a Table D mnemonic,
549 and "0" if the mnemonic is being declared as a Table B mnemonic. Otherwise, the remainder of the FXY
550 number must be all digits, with the next 2 characters (i.e. the "X" component) as a number
551 between 00 and 63, and the final 3 characters (i.e. the "Y" component) as a number between 001
552 and 255. Readers who are more familiar with BUFR will immediately recognize these F, X, and Y
553 values as those that are defined within the
554 [official documentation of the BUFR code form](@ref manual);
555 therefore, by international convention, a mnemonic should not be given an X value between 00 and 47
556 along with a Y value between 001 and 191 unless that mnemonic, when subsequently defined, corresponds
557 exactly to the BUFR descriptor having that same FXY number within the
558 [official WMO master BUFR tables](@ref wmomstab). For example, in our [sample DX BUFR tables file](#bftab),
559 mnemonic "WMOB" is declared with an FXY number of 001001; therefore, it has the exact same text
560 description (i.e. "WMO BLOCK NUMBER") and, when later defined within the last section of the file,
561 the exact same scale factor, reference value, bit width, and units as for FXY number 001001 within
562 the official WMO master BUFR tables. This concept should be somewhat intuitive, but it's obviously very
563 important when the BUFRLIB software is to be used to encode BUFR messages that may potentially be
564 read by other users in other organizations around the world.
565 
566 <br>
567 
568 In looking further at our [sample DX BUFR tables file](#bftab), we see that the lines within the first
569 section each contain a "|" character in columns 1, 12, 21, and 80. Mnemonics are declared, and are
570 left-justified, in columns 3-10, corresponding FXY numbers are assigned in columns 14-19, and the
571 corresponding text description begins in column 23. All of the Table A mnemonics are declared first,
572 followed by all of the Table D mnemonics, followed by all of the Table B mnemonics. Within each set,
573 it is generally a good idea for human-readability purposes to list the mnemonics in ascending order
574 with respect to their FXY number, although this is not required by the BUFRLIB software itself.
575 Likewise, human-readability can usually also be improved by the judicious use of one or more
576 separator lines containing the required "|" character in columns 1, 12, 21, and 80 but without
577 any actual mnemonic declaration; however, again, the use of such separator lines is not required
578 by the software. In fact, the software will simply continue reading lines of the file, one at a time,
579 and looking for new mnemonic declarations, until it reaches a line which does not contain a "|"
580 character in each of columns 1, 12, 21, and 80, at which point it then knows that the first section
581 of the tables file has ended.
582 
583 <br>
584 
585 We mentioned earlier that mnemonics only exist in order to facilitate user interaction with the
586 BUFRLIB software and that, therefore, mnemonics should be as intuitive as possible. We now need
587 to amend that statement slightly, because certain Table A mnemonics do have a special additional
588 function. Specifically, if a Table A mnemonic consists of 8 characters (i.e. the maximum) and if
589 characters 3 through 8 are all digits, then the mnemonic is also used by the software to set the
590 data category and local subcategory within Section 1 of each BUFR message when writing/encoding
591 data subsets corresponding to that mnemonic. In such cases, characters 3 through 5 define the
592 category, and characters 6 through 8 define the subcategory. Therefore, in referring again to our
593 [sample DX BUFR tables file](#bftab) where we've defined three different Table A mnemonics,
594 we've also indicated that, e.g. when we use the software to write/encode data subsets according
595 to the Table A mnemonic "NC002007" (i.e. wind profiler), we want all BUFR messages which contain
596 such data subsets to be encoded as category 2 and local subcategory 7 within Section 1 of the
597 message.
598 
599 <br>
600 
601 Incidentally, even if a Table A mnemonic doesn't meet the above criteria, BUFR message category
602 and local subcategory values will still be set by the software when writing/encoding BUFR data
603 subsets corresponding to that Table A mnemonic. However, in such cases, the category value will
604 be set to the "Y" component (i.e. last 3 digits) of the FXY number corresponding to the mnemonic,
605 and the subcategory value will simply be set to 0. Therefore, it is recommended to use the
606 previous, more-explicit approach when assigning a Table A mnemonic for a data subset to be output,
607 since this approach provides for greater control over the category and subcategory values that
608 will be encoded into Section 1 of the resultant BUFR message. We should also take this opportunity
609 to point out that, when the FXY number corresponding to a Table A mnemonic is actually encoded
610 into a BUFR message, a "3" is actually encoded in place of the "A" which is used in the DX tables
611 file. Put another way, the "A" that appears within the FXY number corresponding to each Table A
612 mnemonic within the tables file is only there so that such mnemonics can be easily distinguished
613 from Table D mnemonics by the software.

616 ### Section 2
617 
618 Now, let's move on to the second section of a DX BUFR tables file. As already stated, this section
619 is used to define, for each Table A and Table D mnemonic that was previously declared in the
620 [first section](#section1), the sequence of Table B (and possibly other Table D) mnemonics which
621 constitutes that mnemonic. The format for this section is a "|" character in columns 1, 12, and 80,
622 with the mnemonic that is being defined listed in columns 3-10 (left-justified), and the sequence
623 of constituent mnemonics beginning in column 14, each one separated from the others by one or more
624 blank characters. For longer sequences, multiple successive lines may be used in a continuation
625 fashion by repeating, within columns 3-10 of each continuation line, the mnemonic being defined.
626 For example, in our [sample DX BUFR tables file](#bftab), the Table D mnemonic MRPSC0 is defined as consisting
627 of the sequence YEAR MNTH DAYS HOUR MINU RPID MRPIDS CLON CLAT SELV CORN, where MRPIDS is itself
628 a Table D mnemonic which is therefore itself defined in a similar manner elsewhere within the
629 section. As was the case with the [first section](#section1), separator lines may be employed
630 within this section in order to improve human-readability, as long as they contain the "|" character
631 that is required to be in columns 1, 12, and 80 for all non-comment lines within this section,
632 and the BUFRLIB software will continue reading lines of the file as though they are part of the
633 second section until it encounters one that does not adhere to this format.
634 
635 <br>
636 
637 At this point, most readers who have taken at least a cursory glance at the
638 [sample DX BUFR tables file](#bftab) will have no doubt begun to wonder about all of the additional
639 punctuation characters and symbols included within the sequence definitions of the second section.
640 We'll now address those concerns by stating that these are replication indicators for the mnemonic(s)
641 in question:
642 
643 | Symbol | Meaning |
644 |--------|---------|
645 | < > | The enclosed mnemonic is replicated using 1-bit delayed replication (either 0 or 1 replications) |
646 | { } | The enclosed mnemonic is replicated using 8-bit delayed replication (between 0 and 255 replications) |
647 | ( ) | The enclosed mnemonic is replicated using 16-bit delayed replication (between 0 and 65535 replications) |
648 | " "n | The enclosed mnemonic is replicated using regular (non-delayed) replication, with a fixed replication factor of n |
649 
650 Examples of most of these cases are shown within the [sample DX BUFR tables file](#bftab), and, through successive
651 application, can lead to the definition of some rather interesting data structures. For example,
652 the Table A mnemonic NC002001, which defines the layout of a data subset of the type "RAWINSONDE - FIXED LAND",
653 consists of the following sequence of Table B and Table D mnemonics:
654 
655 1. UARTM, followed by
656 2. between 0 and 255 replications of RCPTIM, followed by
657 3. between 0 and 255 replications of BID, followed by
658 4. UASID, followed by
659 5. between 0 and 255 replications of UARID, followed by
660 6. between 0 and 255 replications of UARLV, followed by
661 7. either 0 or 1 replications of UASDG, followed by
662 8. between 0 and 255 replications of UARDCS, followed by
663 9. between 0 and 255 replications of RAWRPT, followed by
664 10. between 0 and 255 replications of UACLD, followed by
665 11. either 0 or 1 replications of UAADF, followed by
666 12. WMOB, followed by
667 13. WMOS, followed by
668 14. WMOR
669 
670 where, e.g., the constituent Table D mnemonic UARLV itself consists of the following sequence:
671 
672 1. VSIG, followed by
673 2. QMPR, followed by
674 3. PRLC, followed by
675 4. QMGP, followed by
676 5. either 0 or 1 replications of UAGP07, followed by
677 6. either 0 or 1 replications of UAGP10, followed by
678 7. either 0 or 1 replications of UATMP, followed by
679 8. either 0 or 1 replications of UAWND, followed by
680 9. either 0 or 1 replications of UAWSH
681 
682 and where, in turn, UAGP07, UAGP10, UATMP, etc. are also Table D mnemonics which can themselves be
683 further resolved. So we can even nest certain replication sequences inside of other replication sequences,
684 and, further, via the judicious use of the < > indicator, even turn on/off entire sequences of data values
685 simply and efficiently. An example of this is the UAWSH (i.e. "RADIOSONDE WIND SHEAR DATA") sequence,
686 whose constituent data values are only ever present in a rawinsonde report when a level of maximum wind is
687 being reported (and, even then, not always!). In this case, enclosing the entire sequence within a < >
688 indicator allows the lack of such data within a report level to be noted by the use of a single bit set to "0"
689 (i.e. 0 replications), rather than having to otherwise store the appropriate "missing" value for every
690 constituent data value. Over the course of many data levels within many data subsets within a single BUFR
691 message, this can add up to significant encoding efficiency, and, in turn, the use of less required storage
692 space per BUFR message. So, in summary, the judicious use of replication can even lead to more efficient
693 data storage for certain types of data.

697 Looking back again at the [sample DX BUFR tables file](#bftab), notice how several of the Table D mnemonics
698 such as RCPTIM and BID are used within both the NC001003 and NC002001 data subset types. This brings up
699 a good point; namely, that by logically grouping certain Table B mnemonics together within carefully-constructed
700 Table D sequence mnemonics, such mnemonics can be easily and efficiently re-used within different Table A
701 mnemonic definitions within the same BUFR tables file. In fact, this would be a good time to also point out that,
702 when using the BUFRLIB software, Table D sequence mnemonics are the only types of mnemonics upon which any
703 type of replication may be directly performed. Thus, in particular, if we wish to effect the replication of
704 a single, particular Table B mnemonic, then we must do so by defining a Table D sequence mnemonic whose only
705 constituent is that one particular Table B mnemonic and then replicating the sequence mnemonic.
706 For a specific example of such a situation, take a look at the definition of RAWRPT within the sample file.

718 will no doubt recognize it from Table C of the
719 [official WMO master BUFR tables](@ref wmomstab).
720 In short, the effect of this operator is that, for each Table B mnemonic which follows it within the current
721 sequence, and continuing up until the point in the sequence where a corresponding 201000 operator is reached
722 (and which turns off the effect), ( YYY - 128 ) bits should be added to the bit width that is otherwise defined

724 effect is to change the number of bits occupied by the data value corresponding to that mnemonic within the
725 overall data subset. Thus, for example, the sequence:

734 within the [third section](#section3) of the DX BUFR tables file, and, therefore, that for this occurrence
735 of that mnemonic within the overall data subset, the corresponding data value will occupy ( 12 + 4 ) = 16 bits.

796 IRET would always be set to 100, and if there were less than 100 actual rows of available data values, then

809 extra rows, and our USR output array instead would have looked exactly as it did in our earlier TBLAEX1 example.

813 The fourth and final subset definition TBLAEX4 is even more interesting, not to mention a bit more contrived.
814 Here we have exactly 4 replications of the Table B mnemonics PRLC, GEOP, TMDB and TMDP, but the
815 replication is done via explicitly listing all of the occurrences of these mnemonics within the subset
816 definition, and where the mnemonics appear in slightly different order within each replication. Again, this
817 is a contrived example, but it will serve to better explain how subroutine ufbrep() actually works, and also
818 how it behaves slightly differently from ufbint() and ufbseq() as well as from yet another subroutine ufbstp().
819 First of all, in the case of subroutine ufbrep(), the first listed mnemonic in STR
820 is always treated as a "pivot", meaning that the second dimension of USR (i.e. the number of rows)
821 is always defined by subsequent occurrences of this pivot mnemonic within the overall subset definition, and
822 where any remaining mnemonics within STR are always independently searched for between each successive
823 occurrence of the pivot mnemonic. So in the case of subroutine ufbrep() with STR='PRLC GEOP TMDB TMDP'
824 for TBLAEX4, there will

910 As shown here, the order of the non-pivot mnemonics between each occurrence of the pivot mnemonic PRLC is
911 now very important when using subroutine ufbstp(). Specifically, ufbstp() only ever moves forward from each
912 occurrence of the pivot mnemonic, and for only one non-pivot mnemonic at a time in the same exact order in
913 which they appear in STR. So in this case, in the
914 second row of the output USR array, it searched forward from PRLC for the first occurrence of GEOP, and then
915 only after it found that did it search for the next mnemonic in the string (i.e. TMDB), but only searching
916 forward from GEOP rather than going all the way back to the previous occurrence of the pivot mnemonic PRLC
917 and searching from there. And since it couldn't find any occurrence of TMDB between the location of GEOP
918 within the second replication and the third occurrence of the pivot mnemonic PRLC which signaled
919 the start of the third repliation, then TMDB and TMDP both ended up as "missing" in the second row of the
920 returned USR array.

922 finds GEOP and then begins searching from there for TMDB, which it eventually finds, but not until after it
923 has already stepped past TMDP. And since it can only move forward from the point where it found TMDB, then
924 it never finds TMDP for the third replication before it encounters the fourth and final occurrence of the
925 pivot mnemonic PRLC, which in turn is why TMDP is "missing" in the third row of the returned USR array.

927 finds GEOP and then begins searching from there for TMDB, but by that point it has already stepped past TMDB,
928 so it never finds that nor the subsequent TMDP mnemonic, and therefore both of those values are "missing" in
929 the fourth row of USR as well.

933 Next, take a look at the definitions of the Table D sequence mnemonics TMPSQ3, WNDSQ2, and PCPSQ3; in
934 particular, notice that, within these definitions, there are references to several mnemonics such as .DTHMITM and
935 .DTHMXGS which were not previously-declared within the [first section](#section1) of the table. At first glance,
936 this seems to contradict everything that we previously said about the need to initially declare all mnemonics
937 within the first section; however, upon closer inspection, the reader will notice that there do exist, within
938 the first section, declarations for mnemonics .DTH.... and .DTH..... So, what exactly is going on here? The
939 answer is that each of these is a special mnemonic known as a <i>following-value mnemonic</i>, meaning that,
940 when it is used within a sequence definition, it implies a special relationship with the mnemonic that immediately
941 follows it within the sequence. In fact, this relationship is so special that, when a following-value mnemonic is
942 used within a sequence definition, the .... portion of the mnemonic is replaced with the mnemonic that immediately
943 follows it! For example, when .DTH.... is used within the definition of the Table D sequence mnemonic TMPSQ3, it
944 appears as .DTHMXTM and .DTHMITM because it appears immediately before, respectively, the mnemonics MXTM and MITM.
945 However, when it appears within the definition of PCPSQ3, it appears as .DTHTOPC since it immediately precedes
946 TOPC within that sequence. To be precise, a following-value mnemonic is declared with a "." as the first character,
947 followed by no more than 3 alphanumeric characters as an identifier, followed by 4 more "." characters which must
948 then be replaced with the mnemonic that immediately follows it whenever and wherever it is used within a sequence
949 definition. This is important, because the BUFRLIB software will actually check that the immediately-following
950 mnemonic matches the last 4 characters of the following-value mnemonic and will diagnose an error if it does not.
951 
952 <br>
953 
954 In general, the "following-value" attribute is useful because it allows the same mnemonic to be used repeatedly
955 within the same overall Table A data subset definition in a very intuitive fashion and yet, since each occurrence
956 retains its own unique identification (e.g. .DTHMXTM, .DTHTOPC, etc.), then each one can still be individually
957 accessed independent of the others via subroutine ufbint(). An alternative would be to declare a regular mnemonic
958 such as DTHRFV instead of .DTH.... within the first section of the tables file and then use that mnemonic in all
959 of the same places within the same Table A data subset definition, but then we'd have to use subroutine ufbrep()
960 to access all such values simultaneously, even if we weren't interested in all of them. And we'd also lose the
961 intuitiveness provided by having available, within the mnemonic itself, the name of the mnemonic to which the
962 corresponding value applies.

965 ### Section 3
966 
967 It's now time to move on to the third and final section of a DX BUFR tables file. As we mentioned earlier,
968 this section is used to define the scale factor, reference value, data width, and units for all of the Table B
969 mnemonics that were previously declared in the [first section](#section1). In particular, the reader may recall
970 that the units definition for each Table B mnemonic in turn determines how data values corresponding to that
971 mnemonic are read/written from/to the REAL*8 array USR within BUFRLIB subroutines such as
972 ufbint(), ufbrep() and ufbseq().
973 
974 <br>
975 
976 In looking again at our [sample DX BUFR tables file](#bftab), we see that the format for the third section of such a file
977 is to have our same old, familiar "|" delimiter in columns 1, 12, 19, 33, 39, 66, and 80 of each line.
978 These delimiters, in turn, form the columns for the mnemonic (listed exactly as it was previously within
979 the [first section](#section1)), the scale factor (right-justified from column 17), the reference value
980 (right-justified from column 31), the bit width (right-justified from column 37), and the units (left-justified
981 from column 41). As with the previous two sections, blank separator lines may be employed in order to improve
982 human-readability, and, for the same reason, it's also recommended to list the mnemonics in the same order in
983 which they were declared within the first section, although this is by no means a requirement of the software.
984 However, note that any mnemonic whose corresponding data values are to be treated as character data must have
985 its units listed as "CCITT IA5", which is basically just a formal synonym for ASCII.
